# MT940 Processing — Design Spec

## Goal

Add MT940 bank statement file processing as an alternative transaction source, so the Kruispost Monitor can run without Ponto Connect API access. The user drops an MT940 file in a folder and points the app at it via config.

## Transaction Source Abstraction

Introduce `ITransactionSource` to decouple the orchestration pipeline from the data source:

```csharp
public record TransactionSourceResult(
    List<TransactionRecord> Transactions,
    decimal CurrentBalance,
    string Currency,
    string AccountIdentifier);

public interface ITransactionSource
{
    Task<TransactionSourceResult> FetchTransactionsAsync(DateTimeOffset? since);
}
```

Two implementations:
- **PontoTransactionSource** — wraps existing `IPontoService` (init, find account, sync, fetch)
- **Mt940TransactionSource** — reads and parses an MT940 file from a configured path

## MT940 Parser

A static `Mt940Parser` class: `Mt940Parser.Parse(string fileContent) → Mt940Statement`.

`Mt940Statement` holds:
- Account IBAN (from `:25:` tag)
- Closing balance, amount + currency (from `:62F:` tag)
- List of parsed transactions

### Supported MT940 tags

| Tag | Purpose | Mapping |
|-----|---------|---------|
| `:25:` | Account identification | `AccountIdentifier` (IBAN) |
| `:62F:` | Closing balance | `CurrentBalance` + `Currency` |
| `:61:` | Transaction line | Date, amount, debit/credit indicator |
| `:86:` | Transaction details | Counterpart name, remittance information |

Each `:61:` + `:86:` pair maps to a `TransactionRecord`. Transaction IDs are generated by SHA256-hashing a concatenation of date, amount, debit/credit indicator, counterpart name, and remittance information, then taking the first 16 hex characters. This produces stable, deterministic IDs for state tracking since MT940 has no native unique IDs.

Single-statement files only (one file per run).

### Mt940Statement record

```csharp
public record Mt940Statement(
    string AccountIdentification,
    decimal ClosingBalance,
    string Currency,
    List<TransactionRecord> Transactions);
```

## Configuration

New config section in appsettings.json:

```json
{
  "TransactionSource": {
    "Mode": "Mt940",
    "Mt940": {
      "FilePath": "C:/path/to/statement.sta"
    }
  }
}
```

Settings classes:

```csharp
public class TransactionSourceSettings
{
    public const string SectionName = "TransactionSource";
    public string Mode { get; set; } = "Ponto";  // "Ponto" or "Mt940"
    public Mt940Settings Mt940 { get; set; } = new();
}

public class Mt940Settings
{
    public string FilePath { get; set; } = string.Empty;
}
```

When `Mode` is `"Mt940"`, Ponto settings are not required. When `Mode` is `"Ponto"`, the MT940 file path is ignored.

## Program.cs Changes

The orchestration simplifies to:

1. Load state
2. Resolve `ITransactionSource` based on `TransactionSourceSettings.Mode`
3. Call `source.FetchTransactionsAsync(state.LastRunUtc)`
4. Feed `TransactionSourceResult` into `TransactionMatcher` + `NotificationMessageBuilder` (unchanged)
5. Send notifications
6. Update state (including refresh token if Ponto mode), save

Ponto-specific logic (init, find account, sync, 5s delay, fetch) moves into `PontoTransactionSource.FetchTransactionsAsync()`. The refresh token is exposed via a property on `PontoTransactionSource` so Program.cs can persist it to state.

For MT940 mode, the `since` parameter is ignored (the file contains whatever it contains), and no refresh token update is needed.

## Balance Check

MT940 files contain a closing balance in the `:62F:` tag. This is used for the balance deviation check in `NotificationMessageBuilder`, same as the Ponto account balance.

## File Structure

```
src/Triodos.KruispostMonitor/
  TransactionSource/
    ITransactionSource.cs          # Interface + TransactionSourceResult record
    PontoTransactionSource.cs      # Wraps existing IPontoService
    Mt940TransactionSource.cs      # Reads file, uses Mt940Parser
  Mt940/
    Mt940Parser.cs                 # Pure parser: string → Mt940Statement
    Mt940Statement.cs              # Statement record
  Configuration/
    AppSettings.cs                 # Add TransactionSourceSettings + Mt940Settings

tests/Triodos.KruispostMonitor.Tests/
  Mt940/
    Mt940ParserTests.cs            # Parse real MT940 samples
```

## What Stays Unchanged

- `TransactionRecord`, `MatchResult`, `MatchedPair`, `PossibleMatch`
- `TransactionMatcher` and `StringSimilarity`
- `NotificationMessageBuilder`
- `INotificationSender`, `SlackNotificationSender`, `EmailNotificationSender`
- `IStateStore`, `StateStore`, `RunState`
- `IPontoService`, `PontoService`

## Testing

- `Mt940ParserTests` — parse MT940 content strings, verify correct transaction extraction, balance, account ID, edge cases (empty file, missing tags)
- No tests for `PontoTransactionSource` or `Mt940TransactionSource` (thin wrappers)
